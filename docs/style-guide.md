# Style Guide

## Goal

AI와 사람이 같은 기준으로 코드를 수정할 수 있도록 구현 규칙과 UI 방향을 통일한다.

## General Rules

- 기존 구조를 존중하고 큰 리라이트보다 현재 패턴을 확장한다.
- 추상화는 실제 중복이나 복잡도를 줄일 때만 도입한다.
- 문서화되지 않은 새 라이브러리 추가는 신중하게 판단한다.
- 임시 우회보다 명시적인 에러 처리와 복구 경로를 선호한다.

## Backend Rules

- 흐름은 `Controller -> Service -> Repository`를 기본으로 유지한다.
- DTO, 응답 모델, 요청 모델은 명확히 분리한다.
- 예외는 공통 에러 코드 체계로 연결한다.
- 외부 통신 로직은 서비스에 인라인으로 섞지 않고 분리된 클라이언트로 둔다.
- JPA 연관관계는 필요한 만큼만 유지한다.

## Frontend Rules

- 서버 데이터는 React Query, UI 상태는 Zustand로 구분한다.
- Electron renderer는 브라우저 앱처럼 유지하고 OS 의존 기능은 IPC로 넘긴다.
- 기능 코드는 `features/` 아래에서 응집도를 유지한다.
- API 호출은 기능별 `*.api.ts` 또는 공용 서비스 계층에 둔다.
- 타입 정의 없는 API 응답 처리는 금지한다.

## Security Rules

- `nodeIntegration: false` 유지
- `contextIsolation: true` 유지
- 비밀값과 내부 API 키는 환경변수로만 주입
- 프론트엔드에서 사내 인증 서버 직접 호출 금지
- 인증이 필요한 파일 다운로드나 내부 API는 명시적으로 보호

## Error Handling Rules

- 사용자에게 보여줄 에러와 내부 디버깅용 에러를 구분한다.
- 프론트엔드에서는 실패 시 조용히 무시하지 않는다.
- 네트워크 재시도나 소켓 재연결은 예측 가능한 정책으로 구현한다.

## UI Direction

### Product Tone

- 사내 업무 도구답게 안정적이고 명확해야 한다.
- 장식보다 가독성, 밀도, 상태 전달력을 우선한다.
- 카카오톡 레이아웃 감각을 참고하되 블루 계열 테마를 유지한다.

### Visual Rules

- 좌측 네비게이션, 우측 대화 영역의 2단 구조를 기본으로 삼는다.
- 내 메시지는 우측 정렬과 강조 색상, 상대 메시지는 좌측 정렬과 중립 배경을 사용한다.
- 공지와 외부 알림은 일반 메시지와 다른 시각적 구분을 둔다.
- 읽음 수, 시간, 상태 배지는 작지만 바로 인지 가능해야 한다.

### Color Intent

- Primary: 블루 계열
- Sidebar: 다크 네이비 계열
- Chat background: 밝은 슬레이트 계열
- Notice: 블루 틴트 배경
- External alert: 앰버 또는 경고 계열 강조

## Naming And Code Quality

- 불분명한 축약어보다 도메인 의미가 드러나는 이름을 선호한다.
- `any` 사용은 피한다.
- 테스트가 가능한 로직은 UI 이벤트 핸들러에서 분리한다.
- 주석은 왜 필요한지 설명할 때만 추가한다.
